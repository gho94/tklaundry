# 범위 · 구현 순서

---

## 목표

- 레거시와 **같은 업무**를 차세대에서 처리
- **기초 마스터 4개**를 레거시와 맞춘 뒤 접수·출고·매출 진행
- 문제 생기면 차세대만 끄고 레거시로 계속 (롤백)

## 하지 않는 것

- 고객용 앱/웹, 클라우드, 멀티매장, 외부 API 공개

## 1단계 — 기초 마스터 (접수 전에 끝내기)

레거시 화면·DB 결과가 **같을 때까지**. 접수는 1단계 **이후**.

### ① 회원 (`ComMember`) — 완료

- [x] 로그인 (`user` 응답 · Flutter `authProvider` 적재)
- [x] 자동 로그인 (`shared_preferences` · 앱 재실행 시 login API 재호출)
- [x] 회원가입 (`201` + member 반환 · 로그인 화면 가입 시 즉시 로그인)
- [x] 아이디 중복 확인 (`GET /api/members/exists`)
- [x] 회원 목록·상세·수정·삭제·`UseYn`
- [x] Flutter: 로그인 · 회원 목록(`MemberListPage`) · 등록/수정 다이얼로그

### ② 공통코드 (`ComBaseData`)

- [ ] login 응답 `codes` flat · `codeProvider` 적재
- [ ] flat 목록 조회·트리 UI (앱에서 `pCodeId`로 조립)
- [ ] 등록·수정·삭제 → 캐시 갱신

### ③ 고객 (`ComCustomer`)

- [ ] 목록·검색·등록·수정

### ④ 제품 (`ComProduct`)

- [ ] 목록·검색·등록·수정 (단가·공정코드)

**1단계 완료**: 위 4개가 레거시와 동일. [05_테스트.md](05_테스트.md) 1~4절 통과.

## 2단계 — 접수 · 출고 · 매출 (1단계 후)

순서: **접수 → 출고 → 매출**

### 접수 (`OrderMaster` / `OrderDetail`)

- [ ] 목록 (기간·상태·검색) + 건수·총액 대조
- [ ] 상세, 등록·수정, 상태 변경

### 출고 (`DeliveryMaster` / `DeliveryDetail`)

- [ ] 목록·상세, 출고 처리

### 매출 (`SalesMaster` / `SalesDetail`)

## 병행 운영 (2단계부터)

1. 같은 테이블을 양쪽에서 **동시에 수정하지 않음**
2. 아래 표처럼 **쓰기 담당** 정함
3. 건수·금액 **매일** 레거시와 대조

| 업무 | 레거시 | 차세대 |
|------|:------:|:------:|
| 회원·코드·고객·제품 | W → | W (1단계 후 이동) |
| 접수 | W | W (2단계) |
| 출고 | W | W |
| 매출/수납 | W | W |

**롤백**: API 중지 → 앱 중지 → 레거시로 업무
