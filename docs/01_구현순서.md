# 범위 · 구현 순서

---

## 목표

- 레거시와 **같은 업무**를 차세대에서 처리
- **기초 마스터 4개**를 레거시와 맞춘 뒤 접수·출고·매출 진행
- 문제 생기면 차세대만 끄고 레거시로 계속 (롤백)

## 하지 않는 것

- 고객용 앱/웹, 클라우드, 멀티매장, 외부 API 공개

### ③ 고객 (`ComCustomer`) — 완료

**목록·필터**

- [x] `GET /api/customers?aptCode=` — `aptCode` 생략 시 전체 · 값 있으면 `WHERE AptCode = aptCode` · `aptCode=`(빈 문자열)은 기타
- [x] 정렬: `AptCode, BuildingCode, FloorCode, RoomCode, CustName` (레거시 `SelectTblComCustomer`와 동일)
- [x] Flutter: `CustomerListPage` · 아파트 콤보(전체·기타) · 가상 그리드

**등록**

- [x] 중복: 4코드 **모두 request `!= null`** 일 때만 위치 중복 검사 → `409`
- [x] 저장: 미선택 콤보 `null` → DB `""` (레거시 `BtnSave`와 동일)
- [x] Flutter: 등록 다이얼로그 · 연쇄 콤보 · 이름 자동 조합

- [x] 수정 시 중복 검사 없음 (레거시 `ValidComCustomerData`에서 `comCustomer != null`이면 통과)
- [x] Flutter: 행 더블클릭 수정 · 선택 후 삭제 확인

## 병행 운영 (2단계부터)

1. 같은 테이블을 양쪽에서 **동시에 수정하지 않음**
2. 아래 표처럼 **쓰기 담당** 정함
3. 건수·금액 **매일** 레거시와 대조

| 업무 | 레거시 | 차세대 |
|------|:------:|:------:|
| 회원·코드·고객·제품 | W → | W (1단계 후 이동) |
| 접수 | W | W (2단계) |
| 출고 | W | W |
| 매출/수납 | W | W |

**롤백**: API 중지 → 앱 중지 → 레거시로 업무
