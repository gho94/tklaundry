## 1. Java 코드 구조

### 계층 (위에서 아래로)

```
Controller  →  I*Service(인터페이스)  →  *Service(구현)  →  Mapper  →  DB
```

| 규칙 | 예시 |
|------|------|
| 인터페이스 | `IComMemberService` — 접두어 **I** |
| 구현체 | `ComMemberService` — **Service로 끝남** (`ServiceImpl` 안 씀) |
| 컨트롤러 | 인터페이스만 주입 |

### 패키지 — 레거시 `common` / `sales` 대응

레거시 C#도 크게 **BaseForm(기초)** / **SalesForm(영업)** 으로 나뉜다. Java도 동일하게 2갈래로 맞춘다.

| 레거시 | Java 패키지 | 1·2단계 |
|--------|-------------|---------|
| BaseForm (회원·코드·고객·제품) | `common.*` | 1단계 |
| SalesForm (접수·출고·매출) | `sales.*` | 2단계 |

```
com.tklaundry.api
├── common/                    ← 레거시 BaseForm
│   ├── member/                회원·로그인
│   ├── code/                  공통코드
│   ├── customer/              고객
│   ├── product/               제품
│   ├── config/                필터·인터셉터
│   └── web/                   공통 응답·예외
└── sales/                     ← 레거시 SalesForm (2단계)
    ├── order/                 접수 (Order*)
    ├── delivery/              출고 (Delivery*)
    └── master/                매출·수납 (Sales*)
```

MyBatis XML도 같은 구분:

```
resources/mapper/
├── common/    ComMember, ComBaseData, ComCustomer, ComProduct …
└── sales/     Order, Delivery, Sales …
```

### 코드 예시

```java
// 인터페이스
public interface IComMemberService {
    CommonInfo login(String userId, String password);
    ComMember register(ComMember request);
    List<ComMember> listMembers();
    // ...
}

// 구현
@Service
@RequiredArgsConstructor
public class ComMemberService implements IComMemberService {
    private final ComMemberMapper comMemberMapper;
    private final CommonInfo commonInfo;
}
```

### 메서드 네이밍 (계층별)

레거시 C#은 `SelectTblComMember`, `InsertTblComMember` 식이다. Java는 **계층마다 역할에 맞게** 나눈다.

| 계층 | 규칙 | 회원 예시 |
|------|------|-----------|
| **Controller** | HTTP·화면 동사 | `listMembers`, `getMember`, `register`, `updateMember`, `login` |
| **Service** | 업무 동사 (Controller와 동일) | `listMembers`, `getMember`, `register`, `login` |
| **Mapper** | `select` / `insert` / `update` / `delete` / `count` + 엔티티 | `selectComMemberList`, `insertComMember` |

- Controller·Service에는 `select`를 쓰지 않는다 (`listMembers` 등 업무 동사).
- Mapper는 DB·레거시 SQL 대조용. 신규 CRUD는 `selectComMemberList` 식으로 짓는다.
- **로그인 등 기존 구현**은 레거시 Mapper 이름 그대로 유지 (`getTblComMember`).
- CRUD에 안 맞는 업무(`login` 등)는 Service·Controller에 업무 동사 그대로 둔다.

**레거시 ↔ Mapper 대응 (회원)**

| 레거시 | Mapper |
|--------|--------|
| `SelectTblComMember` | `selectComMemberList` |
| `GetTblComMember` | `getTblComMember` *(기존 유지)* |
| `InsertTblComMember` | `insertComMember` |
| `CheckUserID` | `countComMemberByUserId` |
| 로그인일시 갱신 | `updateComMemberLoginDate` |

**호출 흐름 예**

```
Controller.listMembers() → Service.listMembers() → Mapper.selectComMemberList()
Controller.login()       → Service.login()       → Mapper.getTblComMember()
```

## 2. 프로젝트 설정 (YAML)

> **properties 파일은 쓰지 않습니다.** 설정 파일은 **`application.yml` 하나**만 씁니다.  
> **Spring profile(`local`, `store` 등)은 쓰지 않습니다.**

### 파일 구성

| 파일 | 내용 |
|------|------|
| `application.yml` | DB·포트·MyBatis·로그 전부 |
| `logs/` | 로그 파일 (gitignore) |

DB 비밀번호는 매장 PC 기준이라 `application.yml`에 둘 수 있으나, **git 공개 저장소**라면 환경변수·별도 미커밋 파일로 분리 검토.

### application.yml

```yaml
spring:
  application:
    name: tklaundry-api
  datasource:
    url: jdbc:sqlserver://localhost:1433;databaseName=TKLaundry;encrypt=true;trustServerCertificate=true
    username: sa
    password: "여기에_비밀번호"

server:
  port: 8080

mybatis:
  mapper-locations: classpath:mapper/**/*.xml
  configuration:
    map-underscore-to-camel-case: true

logging:
  level:
    root: INFO
    com.tklaundry.api: INFO
    com.tklaundry.api.common.mapper: DEBUG   # SQL 확인 (필요 시 INFO로)
  pattern:
    console: "%d{HH:mm:ss.SSS} [%thread] %-5level %logger{36} - [traceId=%X{traceId}] %msg%n"
    file: "%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{36} - [traceId=%X{traceId}] %msg%n"
  file:
    name: logs/tklaundry-api.log
  logback:
    rollingpolicy:
      file-name-pattern: logs/tklaundry-api.%d{yyyy-MM-dd}.%i.log
      max-file-size: 10MB
      max-history: 30
      total-size-cap: 500MB
```

- DB·API 주소: **`localhost`** (매장 PC에서 앱·API·DB 같은 머신 전제)
- `127.0.0.1` 대신 `localhost` 사용

### 실행

```powershell
./gradlew bootRun
```

### Flutter 앱 설정

| 파일 | 내용 |
|------|------|
| `lib/core/constants/api_constants.dart` | API 주소 (`http://localhost:8080`) |

## 3. 로그 규칙

### 왜 남기나

- 병행 운영 중 **레거시와 숫자가 안 맞을 때** 원인 추적
- 매장 PC에서 오류 났을 때 **traceId로 로그 파일 검색**
- DB 비밀번호·로그인 비번은 **절대 로그에 안 남김**

### traceId (요청 추적 번호)

1. 앱이 `X-Request-Id` 헤더로 보내거나, 없으면 서버가 UUID 생성
2. 요청 처리 동안 **MDC**에 넣어서 모든 로그 줄에 붙임
3. API 오류 JSON의 `traceId`와 로그 파일의 traceId가 **같아야** 함

로그 예시:
```
14:32:01.123 INFO  ... - [traceId=a1b2c3d4] POST /api/auth/login 200 45ms userId=admin
```

### API 서버 — 남길 것 / 안 남길 것

| 시점 | 남길 내용 | 레벨 |
|------|-----------|------|
| 요청 들어올 때 | method, path, query (비번 제외) | INFO |
| 요청 끝날 때 | HTTP status, 소요시간(ms) | INFO |
| 로그인 성공 | userId | INFO |
| 업무 오류 | code, message, traceId | WARN |
| 예외 | stack trace 전체 | ERROR |
| SQL (로컬만) | MyBatis 실행 쿼리 | DEBUG |

**금지**: `password`, DB URL 비밀번호, 요청 body 통째로

### API 서버 — 구현 위치

```
common/config/
├── TraceIdFilter.java
└── RequestLoggingFilter.java

common/web/
└── GlobalExceptionHandler.java
```

### Flutter 앱

| 시점 | 내용 |
|------|------|
| API 호출 | method, path, status, traceId |
| API 실패 | traceId + 화면에 보여준 메시지 |
| 로그인/로그아웃 | userId (비번 X) |

- 개발: `debugPrint` 또는 `logger` 패키지
- 화면: `TkApiErrorBanner`에 traceId 표시 → 운영자가 API 로그 검색

### 로그 파일

```
backend/logs/
├── tklaundry-api.log                 # 오늘
└── tklaundry-api.2026-06-07.0.log  # 이전 (30일 보관)
```

- `logs/` 폴더는 **gitignore**
- 설정은 `application.yml`의 `logging.*` (위 2절 참고)

## 4. 로그인 · 세션 (Flutter + API)

### authProvider (`AsyncNotifier<AuthUser?>`)

| Provider | 담는 것 |
|----------|---------|
| `authProvider` | 현재 유저 (`userId`, `userName`) |

| 파일 | 역할 |
|------|------|
| `features/auth/data/auth_api.dart` | login · register |
| `features/auth/data/auth_local_storage.dart` | 자동 로그인 저장 (`shared_preferences`) |
| `features/auth/presentation/auth_provider.dart` | 세션 상태 |

**로그인**

1. `POST /api/auth/login` → `{ "user": { userId, userName } }`
2. `authProvider` 저장 → 메인 화면
3. 자동 로그인 체크 시 `userId`·`password` 로컬 저장

**앱 재실행 (자동 로그인)**

1. 저장된 자격증명으로 `POST /api/auth/login` 재호출
2. 성공 시 `authProvider` 복원 → 메인 화면 (로딩 스피너 후)

**신규 회원가입 (로그인 화면)**

1. `POST /api/auth/register` → `201` + member (`password` 제외)
2. 비로그인 가입 시 서버 `CommonInfo.bindLogin` + `LogInDate` 갱신
3. Flutter `signInFromRegister` → `authProvider` 설정 (별도 login 호출 없음)
4. 관리자가 회원 목록에서 등록할 때는 **기존 세션 유지** (자동 전환 없음)

**로그아웃**

- `authProvider` 초기화 + 로컬 저장 삭제

> API는 아직 토큰/세션 헤더 없음. 서버 `CommonInfo`는 싱글톤 — 매장 PC 1대·1명 전제.

### 1-2 공통코드 이후

| Provider | 담는 것 |
|----------|---------|
| `codeProvider` | 공통코드 flat 전체 |

API는 **flat 배열** (트리 JSON 불필요). 레거시 `Dictionary<codeId, …>` 대응:

```
codes[]  →  Map<codeId, CodeItem>
         →  Map<pCodeId, List<CodeItem>>  (콤보박스)
```

1. login 응답 `codes` 또는 `GET /api/codes` → `codeProvider`
2. 공통코드 CRUD 후 → `GET /api/codes` 재조회 → 캐시 갱신

## 5. Flutter 구조

```
lib/
├── core/          theme, router, network
├── shared/        공통 위젯 (Tk*)
├── features/
│   ├── auth/      로그인·세션
```
